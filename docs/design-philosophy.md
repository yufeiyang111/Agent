## 1. 问题：单库单表为什么不行？

### 1.1 一个 MySQL 数据库能存多少数据？

首先看单库的性能瓶颈：

```
MySQL 单库性能上限（经验值）：
  ├── 数据量：建议 ≤ 500GB ~ 1TB
  ├── 单表行数：建议 ≤ 2000 万行（InnoDB）
  ├── 写入 QPS：≤ 5000 ~ 10000（视硬件和查询复杂度）
  ├── 读取 QPS：≤ 10000 ~ 50000
  └── 连接数：建议 ≤ 200 ~ 500
```

**为什么有这些上限？**

**原因①：B+树索引深度**

MySQL InnoDB 用 **B+树** 做索引。当数据量增大时，树的高度增加：

```
数据量 100 万行 → B+树高度 3 层
数据量 1000 万行 → B+树高度 3~4 层
数据量 1 亿行   → B+树高度 4~5 层
数据量 10 亿行  → B+树高度 5~6 层

每增加 1 层 = 多一次磁盘 IO = 查询慢 10ms+
```

**原因②：磁盘 IO 瓶颈**

MySQL 数据最终存在磁盘上。内存（Redis）的速度比磁盘（MySQL）快 1000 倍以上：
- 内存访问：~100ns
- SSD 磁盘：~100us（慢了 1000 倍）
- 机械硬盘：~10ms（慢了 10 万倍）

**原因③：连接数限制**

MySQL 每个连接消耗 ~200KB 内存。500 个连接 = ~100MB。如果高并发时 10000 个请求同时到达，连接池会彻底耗尽。

**原因④：锁竞争**

高并发写同一个表 → 行锁 / 间隙锁 / 表锁 → 请求排队 → 性能雪崩。

### 1.2 做个简单计算

```
假设：
  - 每天订单量：500 万
  - 一年订单量：500万 × 365 = 18.25 亿
  - 每张订单记录约 500 字节
  - 一年数据量：18.25亿 × 500B ≈ 91GB

单表存 18 亿行数据：
  × 索引深度 6 层 → 查询慢
  × 备份耗时 10+ 小时
  × ALTER TABLE 加一列要跑几小时
  × 磁盘 IO 成为瓶颈

结论：不加分库分表，系统撑不过半年。
```

### 1.3 分库分表的解决思路

```
分库分表的本质：分而治之（Divide and Conquer）

单库 = 1 个人干 100 人的活
分库 = 100 个人各干 1 人的活

单表 100 亿行 → 分成 512 张表 → 每张表 ≈ 2000 万行
                              ↓
                    每张表 B+树高度 3 层 → 查询飞快
```

## 2. 分库分表：数据再多也不怕

### 2.1 什么是分库分表？

```
分库：把数据分散到多个数据库实例
  - DB_0, DB_1, DB_2, DB_3 ...
  - 每个库跑在不同的服务器上
  - 突破单机硬件上限

分表：把一个表拆成多个表
  - user_0, user_1, user_2 ... user_31
  - 每张表数据量大幅减少
  - 索引深度降低，查询更快
```

### 2.2 水平分片 vs 垂直分片

**垂直分片（Vertical Sharding）**：

```
按业务模块拆分：
  ┌─────────────────────────────────────┐
  │          一个超大数据库               │
  ├─────────┬─────────┬────────┬────────┤
  │ user 表  │ order 表 │ product│ review │  →  拆成独立库
  │         │         │        │        │
  └─────────┴─────────┴────────┴────────┘
                   ↓
  ┌─────────┐  ┌─────────┐  ┌────────┐  ┌────────┐
  │ 用户库    │  │ 订单库   │  │ 商品库  │  │ 评价库  │
  │ user_db │  │order_db │  │prod_db │  │ rv_db  │
  └─────────┘  └─────────┘  └────────┘  └────────┘
```

**水平分片（Horizontal Sharding）**：

```
按数据行拆分（垂直分片后，每个库再水平拆分）：
  ┌──────────────┐
  │  订单库(order_db, 8个实例)    │
  ├──────┬──────┬──────┬──────┤
  │库0   │库1   │库2   │...库7 │
  ├──────┼──────┼──────┼──────┤
  │表0~63│表0~63│表0~63│表0~63│  → 共 512 张表
  └──────┴──────┴──────┴──────┘
  
  每个库 ≈ 1/8 的数据
  每张表 ≈ 1/512 的数据
```

### 2.3 为什么要垂直 + 水平两层分片？

```
先垂直分片（按业务拆库）：
  ✅ 各业务互不影响（订单库慢不影响商品库）
  ✅ 可针对不同业务选择不同硬件（订单库更强，营销库可弱）
  ✅ 方便按业务拆分微服务

再水平分片（每业务再拆多库多表）：
  ✅ 进一步降低单表数据量
  ✅ 提升写入并行度
  ✅ 方便扩容
```

### 2.4 多少数据量应该分库分表？

```
经验法则（来自一线大厂）：

  单表行数        → 建议操作
  < 500 万        → 不用分，优化索引就行
  500万 ~ 2000万  → 考虑分表（每表 ≤ 1000万）
  2000万 ~ 1亿    → 必须分表
  > 1亿           → 分库 + 分表

  单库数据量       → 建议操作
  < 100GB         → 不用分
  100GB ~ 500GB   → 考虑分库
  > 500GB         → 必须分库
```

## 3. 分片键：数据该怎么分配？

### 3.1 分片键是最重要的决策

分片键 = 按哪个字段来分配数据到哪个分片。

```
❌ 选错分片键的后果：
   - 热点数据集中到一个分片（其他分片空闲）
   - 大部分查询都要跨分片（性能极差）
   - 分片扩容要重新分配数据

✅ 选对分片键的好处：
   - 数据均匀分布到所有分片
   - 90%+ 的查询只查一个分片
   - 扩容相对简单
```

### 3.2 我们的分片键选择

| 表 | 分片键 | 为什么选这个？ |
|----|--------|--------------|
| user | user_id | 用户登录后所有操作都带 user_id |
| order | user_id | "我的订单"查询最频繁，按用户ID分片可以一次查到所有订单 |
| spu | spu_id | 商品查询按商品ID |
| review | spu_id | 查看评价时以商品为单位 |

### 3.3 为什么订单按 user_id 分片？（关键问题）

这个问题是电商分片设计的核心。

**两种方案对比：**

```
方案 A：按 order_id 分片
  - 优点：知道订单号就能定位分片
  - 缺点：用户查看"我的订单"时，订单分散在所有分片中，需要"广播查询"所有分片
  - 广播查询 ≈ 查询 8 个库 = 8 倍延时，还可能有重复/漏数据

方案 B：按 user_id 分片（我们的方案）
  - 优点：用户的所有订单都在同一个分片中，"我的订单"查一次就是一个分片
  - 缺点：知道订单号但不知道 user_id 时，无法定位分片
  
我们选择方案 B，因为：
  - "我的订单"查询量占 90% 以上（用户最常用）
  - 通过订单号查的场景只占不到 10%（客服使用）
  - 订单号查不到 user_id 的问题，用"订单号→用户ID映射表"解决
```

### 3.4 分片算法：取模 vs 哈希

```
取模（MOD）：
  db_idx = user_id % 8
    → 简单直观，推荐
  
一致性哈希（Consistent Hashing）：
  db_idx = hash(user_id) on ring
    → 扩容时只需迁移 1/n 的数据
    → 但实现复杂，不适合我们的场景

为什么选择取模？
  - 我们前期确定了 8 个库，后期扩容可以通过"双写"方案平滑迁移
  - 取模算法计算最快（位运算），不增加额外延迟
  - 实现简单，ShardingSphere 原生支持
```

### 3.5 避免"跨分片查询"

跨分片查询是指一次查询需要扫描多个分片，这是分库分表最想避免的。

```
跨分片查询的例子：
  SELECT * FROM order WHERE create_time > '2024-01-01'
  → 不知道 user_id → 需要查 8 个库再聚合 → 慢 8 倍

避免跨分片查询的 3 个方法：
  ① 查询条件必须带分片键（where user_id = ?）
  ② 无法带分片键的，走 ES 搜索引擎
  ③ 后台统计查 ClickHouse（不是查 MySQL）
```

## 4. 分布式 ID：雪花算法详解

### 4.1 为什么不能用数据库自增 ID？

```
场景：订单表分成了 512 张表

如果每张表都用 AUTO_INCREMENT：
  order_0: 1, 2, 3, 4, 5 ...
  order_1: 1, 2, 3, 4, 5 ...
  order_2: 1, 2, 3, 4, 5 ...
  
  问题：不同表的 ID 冲突了！没法做全局唯一标识。

如果用 UUID：
  order_no = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'
  ✅ 全局唯一
  ❌ 32 位字符串 → 占用空间大（比 BIGINT 大 4 倍）
  ❌ 无序 → B+树插入时频繁页分裂 → 性能差
  ❌ 作为主键时，索引变大，查询变慢
```

### 4.2 雪花算法结构详解

雪花算法生成的 ID 是 64 位整数（BIGINT），分成 4 段：

```
 0 | 0000000000 0000000000 0000000000 0000000000 00 | 0000000000 00 | 0000000000 00
 ↑                        ↑                              ↑                ↑
符号位                 41 位时间戳                     10 位机器 ID     12 位序列号
(1 bit)               (毫秒级,可用 69 年)              (支持 1024 台)   (每毫秒 4096 个)
```

**逐段解释：**

**① 符号位（1 bit）：**
- 始终为 0，表示正数
- 为什么？Java 的 long 类型第一位是符号位

**② 时间戳（41 bit）：**
- 存储从某个"纪元"开始的毫秒数
- 41 bit 最大 = 2^41 = 2,199,023,255,552 毫秒 ≈ 69.7 年
- 假设纪元 = 2024-01-01，可用到 2093 年
- `作用：ID 按时间递增排序`

**③ 机器 ID（10 bit）：**
- 高 5 bit = 数据中心 ID（最多 32 个数据中心）
- 低 5 bit = 机器 ID（每个数据中心最多 32 台机器）
- 总共支持 1024 台机器
- `作用：不同机器生成的 ID 不会冲突`

**④ 序列号（12 bit）：**
- 同一毫秒内从 0 开始递增
- 12 bit = 最多 4096 个/毫秒/机器
- 也就是说一台机器一毫秒能生成 4096 个 ID
- 1024 台机器每秒 = 1024 × 1000 × 4096 ≈ 42 亿个 ID/秒
- `作用：同一毫秒内多个 ID 不冲突`

### 4.3 雪花算法三大优点

```
优点 ①：全局唯一
  时间戳 + 机器 ID + 序列号 = 不可能重复

优点 ②：趋势递增
  时间戳在高位，所以后生成的 ID 比先生成的 ID 大
  → B+树插入时基本在末尾 → 页分裂少 → 性能好

优点 ③：高性能
  纯内存计算，不依赖数据库
  单机单毫秒 4096 个 = 每秒 400 万个
```

## 5. 冷热分离：省钱又提速

### 5.1 什么是冷热数据？

```
热数据（Hot Data）：经常被访问的数据
  例：最近 1 个月的订单 → 用户频繁查看物流/状态
  特点：访问频率高，数据量少（占总量 ~10%）

冷数据（Cold Data）：很少被访问的历史数据
  例：1 年前的已完结订单 → 基本不会被查看
  特点：访问频率低，数据量大（占总量 ~90%）
```

### 5.2 为什么需要冷热分离？

**原因①：性能**

```
如果所有数据都在同一个 MySQL 表里：
  - 全表扫描要扫描所有数据（包括 90% 的冷数据）
  - 索引因为数据量大而深度增加
  - 缓冲池（Buffer Pool）里被冷数据污染
  
类比：如果你在找昨天的一份文件，但你桌上堆了过去 10 年的文件，找起来会非常慢。
```

**原因②：成本**

```
存储成本对比：
  ┌──────────┬────────────┬──────────────┐
  │ 存储类型  │ 性能        │ 每GB/月成本   │
  ├──────────┼────────────┼──────────────┤
  │ NVMe SSD │ 极快(5us)  │ ~2 元         │
  │ SATA SSD │ 快(100us)  │ ~0.6 元       │
  │ HDD      │ 慢(10ms)   │ ~0.2 元       │
  │ ClickHouse│ 分析快(列存)│ ~0.4 元(含压缩)│
  └──────────┴────────────┴──────────────┘

如果 100TB 数据全用 NVMe SSD 存：
  100TB × 2元/GB/月 = 204,800 元/月

如果热数据（10TB）存 SSD，冷数据（90TB）存 ClickHouse：
  10TB × 2元 + 90TB × 0.4元 = 20,480 + 36,864 = 57,344 元/月
  每月节省约 15 万元！
```

**原因③：查询效率**

```
MySQL 的 Buffer Pool 是有限的内存空间：
  - Buffer Pool = 数据库的"工作台"
  - 如果工作台被 90% 的冷数据占据 → 热数据反而没位置
  - 热数据查询 → 磁盘 IO → 变慢

冷热分离后：
  - Buffer Pool 里全是热数据
  - 热数据查询基本都在内存中完成（速度提升 1000 倍）
```

### 5.3 冷热分离的实施方案

```
           ┌──────────────┐
           │   写入入口     │
           └──────┬───────┘
                  │
          ┌───────▼───────┐
          │  MySQL 热库     │  ← 所有数据先写热库
          │  (NVMe SSD)    │
          └───────┬───────┘
                  │
          ┌───────▼───────┐
          │  每日 02:00     │
          │  定时迁移任务    │
          │  (批量处理)     │
          └───────┬───────┘
                  │
          ┌───────▼───────┐
          │  ClickHouse 冷库 │  ← 列式存储，压缩率高
          │  (SATA HDD)     │  ← 老数据写在便宜的硬盘上
          └───────────────┘
```

### 5.4 冷热分离的业务规则

以订单为例：

```sql
-- 判断条件
WHERE order_status IN ('已完成', '已关闭', '已退款')
  AND create_time < DATE_SUB(NOW(), INTERVAL 6 MONTH)
```

为什么是 6 个月？
- 电商平台 90% 的订单查询集中在 3 个月内
- 6 个月后的订单几乎不会再被操作（退换货期限已过）
- 留 6 个月的安全缓冲

### 5.5 用户查询冷数据的体验

```
用户查"我的订单"：
  → 默认查 MySQL 热库（最近 6 个月）
  → 99% 的用户在这步就完成了
  
如果用户想看更早的订单：
  → 页面底部有一个"查询历史订单"按钮
  → 点击后显示"正在查询历史数据..."
  → 异步查 ClickHouse 冷库
  → 返回结果

这样设计的好处：
  - 99% 的查询快（只查热库）
  - 冷库查询虽然慢一点，但极少用到
  - 用户感知不到"冷热分离"的存在
```

## 6. 缓存策略：把数据放在最快的地方

### 6.1 计算机存储金字塔

```
                  CPU 寄存器：~1ns（几 KB）
                        ↑
                 L1/L2/L3 缓存：~10ns（几 MB）
                        ↑
                  内存（RAM）：~100ns（几 GB ~ TB）
                        ↑
                  SSD 磁盘：~100us（几 TB）
                        ↑
                  机械硬盘：~10ms（十 TB）
                        ↑
                    网络/磁带：更慢

差多少？
  内存访问比 SSD 快 ~1000 倍
  SSD 比机械硬盘快 ~100 倍
```

### 6.2 多级缓存架构

本项目的缓存从快到慢分 4 级：

```
① 浏览器缓存（最快）
   资源文件（图片/JS/CSS）存浏览器本地
   HTTP 缓存头（Cache-Control, ETag）
   访问时间：~1ms（命中情况下）

② CDN 缓存（近）
   静态资源分发到离用户最近的 CDN 节点
   图片/视频/CDN 全球加速
   访问时间：~10ms

③ Redis 缓存（快）
   热数据存在内存中
   商品详情、用户 Session、库存
   访问时间：~1ms

④ 本地缓存（Caffeine）
   每个应用进程内缓存
   分类树、配置数据
   访问时间：~0.1ms（比 Redis 还快，因为不走网络）
```

### 6.3 缓存什么数据？

```
必须缓存（命中率极高，缓存效果显著）：
  ✅ 商品详情 → 所有用户访问最多的数据
  ✅ 商品分类树 → 所有页面都需要
  ✅ 用户 Session → 每次请求都需要验证
  ✅ 热门商品排行榜 → 首页必备
  ✅ SKU 库存 → 超高并发

选择性缓存（根据业务需要）：
  ⚠️ 购物车 → 读多写也多，注意一致性问题
  ⚠️ 用户信息 → 更新频率低，适合缓存
  
不缓存（收益不大）：
  ❌ 订单详情 → 每个用户只查自己的，缓存命中率低
  ❌ 搜索列表 → 千人千面，缓存命中率极低
```

### 6.4 缓存三大坑及解决方案

#### 坑①：缓存穿透

```
现象：查询一个"一定不存在"的数据
场景：恶意攻击者查询一个不存在的商品 ID

流程：
  查 Redis → 没有（Key 不存在）
  查 MySQL → 没有（数据不存在）
  每次都这样 → MySQL 被无效查询打垮

解决方案：
  ① 布隆过滤器（Bloom Filter）
     - 预先在内存中标记所有存在的 Key
     - 查询前先问 Bloom Filter："这个 Key 存在吗？"
     - 回答"不存在"就直接返回
     - 误判率极低（0.1%）
  
  ② 缓存空结果
     - 查 MySQL 发现不存在时，在 Redis 存 null（TTL 30s）
     - 下次同样的查询直接返回 null
     - 防止恶意攻击重复查询
```

#### 坑②：缓存击穿

```
现象：一个热点 Key 过期瞬间，大量请求同时打到 MySQL
场景：秒杀商品详情突然过期，10 万人同时刷新

流程：
  Key 过期 → 10 万请求同时来
  → 10 万请求都发现缓存没有
  → 10 万请求同时打 MySQL
  → MySQL 被打垮

解决方案：互斥锁
  - 只有一个请求能拿到锁去查 MySQL
  - 其他请求等待或返回旧值
  - 伪代码：
    data = redis.get(key)
    if data is None:
        if redis.setnx("lock:" + key, 1):  # 只有第一个能拿到锁
            data = db.query(key)
            redis.set(key, data, ttl)
            redis.delete("lock:" + key)
        else:
            Thread.sleep(50)  # 等 50ms
            data = redis.get(key)  # 拿第一个线程重建的数据
```

#### 坑③：缓存雪崩

```
现象：大量 Key 同时过期，或者 Redis 宕机
场景：所有商品缓存设了同样的过期时间，到点时集体失效

解决方案：
  ① 过期时间加随机值
     TTL = 3600 + random(0, 600)  → 不再是同一秒过期
  
  ② 本地缓存兜底（Caffeine）
     即使 Redis 全挂了，每个应用的本地缓存还能撑一会
  
  ③ Redis 高可用
     主从 + 哨兵 / Cluster 集群模式，单点故障自动切换
```

### 6.5 缓存一致性问题（最关键的问题）

```
问题：MySQL 数据改了，但 Redis 还是旧数据 → 脏读

场景：
  Step 1：用户 A 看商品价格 → 缓存 = 100 元
  Step 2：商家改价 → MySQL = 80 元
  Step 3：用户 B 看商品价格 → 缓存还是 100 元（脏数据！）

解决方案：旁路缓存（Cache Aside Pattern）
  更新数据时：
    ✅ 先更新 MySQL
    ✅ 再删除 Redis Key（不是更新！）
    ❌ 不要先删缓存再更新 MySQL（并发时有问题）

  读取数据时：
    ✅ 先查 Redis
    ✅ 没有 → 查 MySQL → 写入 Redis → 返回

为什么是"删除"而不是"更新"缓存？
  - "更新"缓存是个复杂操作（可能需要组装数据）
  - "删除"是原子操作（DEL key）
  - 下次读取时自然按需要重建缓存
  - 懒惰策略 = 读取时才重建，用多少建多少
```

## 7. 读写分离：分摊数据库压力

### 7.1 什么是读写分离？

```
日常访问中，数据库读和写的比例大概是：

  读：写 ≈ 9：1（甚至更高）

  比如浏览商品页：
  - 1 次写入（加购物车/下单）
  - 20 次读取（查看商品信息/图片/评价）
  
读写分离 = 把读和写交给不同的数据库实例处理：

          ┌──────────┐
          │   写入    │
          │  主库     │ ← 只有这个库能写入
          └────┬─────┘
               │ binlog 同步（异步/半同步）
          ┌────┴─────┐
          │  从库 1   │ ← 读取本地副本
          ├──────────┤
          │  从库 2   │ ← 读取本地副本
          ├──────────┤
          │  从库 3   │ ← 读取本地副本
          └──────────┘
```

### 7.2 为什么需要读写分离？

```
单库场景（不分离）：
  - 1 个慢查询 SELECT 占满了 CPU
  - 所有 INSERT 跟着一起排队
  - 用户下不了单！→ 直接损失收入

读写分离后：
  - 慢查询打到了从库
  - 主库专心处理写入
  - 下单不受影响
```

### 7.3 我们系统的主从配置

| 业务库 | 主库 | 从库 | 说明 |
|--------|------|------|------|
| 用户库 | 1 | 2 | 读需求中等 |
| 商品库 | 1 | 2 | 读需求高（商品页访问量大） |
| 订单库 | 1 | 2 | 读需求高 + 写需求也高 |
| 评价库 | 1 | 2 | 读需求高 |
| 营销库 | 1 | 1 | 读写都不高 |

### 7.4 读写分离的延时问题

```
问题：主从同步有延迟（通常 1ms ~ 100ms）

场景：
  用户下单 → 主库写入成功 → 立即查询订单 → 从库还没同步到
  → "我的订单"看不到刚刚下的单 → 用户体验极差！

解决方案：
  ① 强制读主库（对一致性要求高的数据）
     下单后 1 秒内的订单查询强制走主库
  
  ② 缓存标记
     下单后设置 Redis Key "user:1001:new_order"
     查询时发现有这个 Key → 走主库
  
  ③ 业务层面容忍
     友好提示"订单已提交，正在处理中..."
```

## 8. ES 搜索：为什么不用 MySQL LIKE？

### 8.1 MySQL LIKE 的局限性

```sql
-- 用户搜索"iPhone 15"
SELECT * FROM spu WHERE spu_name LIKE '%iPhone 15%';
```

**为什么 LIKE 慢？**

```
MySQL LIKE '%关键词%':
  - 最左匹配规则被破坏（%在最前面）
  - 索引失效 → 全表扫描
  - 1000 万商品 → 扫描 1000 万行 → 耗时几秒甚至几十秒

即使改用 LIKE '关键词%'（没有前导%）：
  - 只能用前缀匹配
  - 搜"iPhone"搜不到"Apple iPhone 15 Pro Max"
  - 不支持分词搜索
```

### 8.2 数据库与搜索引擎对比

| 能力 | MySQL | Elasticsearch |
|------|-------|--------------|
| 全文搜索 | ❌ 差（只有 LIKE） | ✅ 强（倒排索引） |
| 分词 | ❌ 不支持中文分词 | ✅ ik_smart/ik_max_word 分词 |
| 拼音搜索 | ❌ 不支持 | ✅ 拼音插件 |
| 同义词 | ❌ 不支持 | ✅ 同义词词典 |
| 纠错 | ❌ 不支持 | ✅ 拼写纠错 suggest |
| 搜索排序 | ❌ 只能按字段排序 | ✅ 相关性评分（TF-IDF/BM25） |
| 高亮 | ❌ 不支持 | ✅ 搜索结果高亮 |
| 聚合统计 | ⚠️ GROUP BY 慢 | ✅ aggregation 极快 |
| 性能（千万级） | ❌ 秒级 | ✅ 毫秒级 |

### 8.3 倒排索引的原理（面试必考）

```
正排索引（MySQL 的 B+树）：
  文档1: "iPhone 15 手机壳"
  文档2: "华为 Mate 60 手机"
  文档3: "iPhone 数据线"

  搜索"手机" → 扫描每个文档 → 匹配到文档 1 和 2
  → O(n) 时间复杂度

倒排索引（Elasticsearch）：
  term → 包含它的文档列表
  手机 → [文档1, 文档2]
  iPhone → [文档1, 文档3]
  华为 → [文档2]
  数据线 → [文档3]

  搜索"手机" → 直接查词典 → 得到 [文档1, 文档2]
  → O(1) 时间复杂度

倒排 = 反过来：记录"每个词出现在哪些文档"
         而不是"每个文档有哪些词"
```

### 8.4 数据同步方案

```
MySQL → ES 数据同步的 3 种方案：

方案 A：Canal 监听 Binlog（推荐）
  MySQL Binlog → Canal → MQ → ES
  实时性：秒级
  优点：对业务代码无侵入
  缺点：需要维护 Canal 服务

方案 B：业务双写
  代码里先写 MySQL，再写 ES
  实时性：毫秒级
  优点：实现简单
  缺点：耦合度高，可能不一致

方案 C：定时全量同步（兜底）
  每天跑批同步全量数据
  做方案 A/B 之外的对账和修复
```

## 9. 秒杀系统：最高并发场景的独立设计

### 9.1 秒杀为什么难？

```
日常访问（无促销）：
  - 浏览商品：5000 QPS
  - 下单：500 QPS
  - 数据库负担：小

秒杀瞬间：
  - 抢购请求：100,000+ QPS（2019 年小米秒杀峰值 ≈ 150万 QPS）
  - 所有人抢同一个商品 → 热点集中
  - 请求是日常的 200 倍 → 直接打数据库等于自杀
```

### 9.2 秒杀的应对策略——逐层过滤

```
每一层都干掉大部分请求，只有极少数请求真正走到数据库。

第一层：CDN + 静态化
  - 秒杀页面纯静态 HTML，直接由 CDN 返回
  - 不经过任何服务器 → 干掉 90% 的网络开销

第二层：Nginx 限流
  - 单 IP 每秒最多 5 次请求
  - 超出直接返回"请求太频繁"
  - 无人机刷单直接被过滤

第三层：Sentinel 接口限流
  - 秒杀接口限制总 QPS = 10000
  - 超出直接返回"排队中，请稍候"

第四层：RocketMQ 削峰
  - 10000 QPS 请求先入 MQ 队列
  - 后端慢慢消费（比如每秒处理 1000 个）
  - 峰被削平了

第五层：Redis LUA 扣库存
  - Redis 单线程 + LUA 原子性
  - 每秒 10 万 + 的原子操作
  - 确认库存 > 0 才扣减

第六层：异步下单
  - 扣库存成功 → 发 MQ 消息 → 异步创建订单
  - 扣库存失败 → 返回"抢光了"

最终真正落到 MySQL 的写入量：
  100,000 请求 → 限流后剩 10,000 → MQ 消费每秒 1000
  只有 1% 的请求最终写入数据库
```

### 9.3 秒杀库存为什么要独立？

```
不要把秒杀库存和日常库存混在一起！

日常库存：sku_stock.available_stock
  优点：统一管理
  缺点：秒杀时超高并发锁住这一行，日常下单也受影响

秒杀库存：seckill_product.seckill_stock（独立字段）
  优点：秒杀和日常互不影响
  缺点：需要额外开发维护

本质：隔离热点——不让秒杀影响正常业务
```

### 9.4 限购的重要性

```
不限购的后果：
  - 黄牛用机器脚本大量抢购
  - 普通用户抢不到
  - 商品全被高价转卖

限购方式：
  每个用户在 Redis 中的记录：
  seckill:user:1001:{activity_id} = 已购买数量

  LUA 脚本中同时校验：
  if bought >= limit → 返回"已达限购数量"
```

## 10. 消息队列削峰：用时间换空间

### 10.1 为什么需要消息队列？

```
没有 MQ 的场景：
  下单请求直接写入数据库：
  ┌────────┐     ┌────────┐
  │ 订单服务  │────▶│ 数据库   │
  └────────┘     └────────┘
  
  如果 10000 个请求同时到达 → 数据库连接打满 → 崩溃

有 MQ 的场景：
  下单请求先发到 MQ，再慢慢消费：
  ┌────────┐     ┌────────┐     ┌────────┐
  │ 订单服务  │────▶│   MQ   │────▶│ 数据库   │
  └────────┘     └────────┘     └────────┘
  10000请求/秒    排队中        1000请求/秒

  MQ 就像一个"缓冲池"，让峰值流量平滑下来。
```

### 10.2 消息队列的 3 个核心作用

```
① 削峰填谷
  瞬时高峰 → MQ 缓冲 → 下游稳定处理
  例：秒杀 10万 QPS → MQ 削峰 → 数据库 1000 QPS

② 异步解耦
  订单创建成功后，不需要同步等所有事情做完：
  ┌──────────┐
  │ 创建订单    ├─────▶ 同步返回"成功"
  └────┬─────┘
       │ 发 MQ 消息
  ┌────▼─────┐  ┌────▼─────┐  ┌────▼─────┐
  │ 扣库存     │  │ 发通知    │  │ 同步 ES  │
  │ (异步)    │  │ (异步)   │  │ (异步)   │
  └──────────┘  └──────────┘  └──────────┘

③ 流量控制
  下游服务压力大时可以"限流消费"（慢慢处理）
  下游挂了时消息会暂存 MQ，恢复后继续处理
```

### 10.3 本系统使用 MQ 的场景

| 发送方 | 主题 | 消费方 | 用途 |
|--------|------|--------|------|
| 订单服务 | order_create_topic | 库存服务 | 异步扣库存 |
| 订单服务 | order_create_topic | 搜索服务 | 商品同步 ES |
| 支付回调 | payment_callback_topic | 订单服务 | 处理支付结果 |
| 后台管理 | data_sync_topic | 搜索服务 | 数据同步 ES |
| 定时任务 | data_archive_topic | 归档服务 | 冷数据迁移 |

## 11. 订单按买家分片：核心设计决策

### 11.1 为什么要重点讨论这个？

分片键的选择是分库分表**最重要且不可逆**的决策。订单的分片键选择尤其关键，因为：
- 订单是数据量最大的表
- 订单涉及最多的查询场景
- 分片键错了后面极难修改

### 11.2 三种分片方案详细对比

```
方案一：按订单 ID（order_id）分片
  ├── 优点：知道订单号直接定位，方便客服查询
  └── 缺点：
       用户查"我的订单" → 不知道分布在哪个分片 → 广播查询所有分片
       卖家查"店铺订单" → 还得建映射表或 ES

方案二：按买家 ID（user_id）分片  ← 我们的方案
  ├── 优点：
  │     用户查"我的订单" → 只查一个分片（极快）
  │     同一个用户的所有订单数据在一起 → 方便做用户分析
  └── 缺点：
       需要订单号 → 用户ID的映射

方案三：按卖家 ID（seller_id）分片
  ├── 优点：卖家查订单很快
  └── 缺点：用户查订单要广播；平台模式订单共享复杂

选择方案二的原因（数据说话）：
  电商系统中，查询分布大约是：
  - 用户查自己的订单：90%
  - 按订单号精确查询：8%
  - 卖家/客服查：2%
  
  90% 的场景应该是最快的 → 按 user_id 分片
```

### 11.3 订单号到用户 ID 的映射方案

解决了方案二的唯一缺点：

```
方案 A：订单号编码法（推荐，无额外存储）
  订单号 = 业务码(2位) + 时间戳(8位) + 用户ID末4位 + 序列号(4位)
  
  例：OD2024010110010001
      解析：业务码=OD, 时间=20240101, 用户ID末4位=1001, 序列=0001
      通过订单号直接知道 user_id 末4位 → 缩小分片范围

方案 B：映射表（兜底方案）
  order_no_mapping 表：order_no → user_id
  先查映射表得到 user_id，再路由到分片
```

## 12. 数据冗余：用空间换时间

### 12.1 什么是合理冗余？

```
电商中两个典型的冗余设计：

① 订单商品快照（最重要的冗余）
  用户下单时，把商品信息"拍照"存到订单中：
  order_item.sku_name     = "iPhone 15 Pro Max"
  order_item.sku_image    = "https://..."
  order_item.price        = 8999.00
  order_item.attrs_snapshot = {"颜色":"钛金属原色","容量":"256GB"}

  为什么这样做？
  一年后商家修改了商品名称、换了图片、调整了价格
  但用户的历史订单中，商品信息仍然是下单时的样子
  
  如果不冗余：
  order_item 关联 sku_id → 但 SKU 信息已经变了
  → 历史订单显示错误的商品名和价格！

② 地址快照（同样重要）
  order 表中有 consignee_name, receiver_phone, province, city...
  
  为什么在 order 表里存一遍地址？
  用户删了收货地址 → 订单还需要发货地址
  地址信息是下单时的"合同" → 必须保持不变
  
  如果不冗余：
  order -> user_address_id -> address
  用户删了地址 → 订单找不到收货地址 → 无法发货！
```

### 12.2 冗余 vs 三范式的平衡

```
三范式（3NF）：理论上"完美"的设计
  order → order_item → sku → spu
  优点：数据不冗余，更新一处生效
  缺点：查询要 JOIN 3 张表 → 慢（跨分片不能 JOIN）

反范式设计（我们的设计）：
  order 表中有商品名、价格、地址等信息
  优点：不 JOIN，查询快
  缺点：数据冗余，商家改商品名后历史订单不受影响

核心原则：
  ✅ 冗余不可变的历史快照
  ✅ 冗余频繁查询的字段（减少 JOIN）
  ❌ 不要冗余频繁变化的数据（如库存）
```

### 12.3 分库分表下为什么不能 JOIN？

```
没有分库时 JOIN 是这样的：
  SELECT * FROM order o
  JOIN order_item oi ON o.order_no = oi.order_no
  → 在同一个数据库中直接关联

分库分表后：
  order_0 ~ order_7（8 个不同实例）
  order_item_0 ~ order_item_7（对应 8 个实例）
  
  如果要 JOIN：
  → 需要从 8 个库分别查出数据
  → 在应用层内存中合并
  → 网络开销 8 倍 + 内存消耗巨大
  → 性能极差！

所以我们才要：
  ① 在 order 表中冗余 order_item 的常用字段（商品名、价格）
  ② 大部分查询不 JOIN
  ③ 真需要完整 JOIN 的场景 → 在应用层分步查询
```

## 13. 常见问题 FAQ

### Q1：分库分表后还能用事务吗？

```
本地事务（同一个分片内）：✅ 完全支持
  因为数据在同一个 MySQL 实例中，ACID 和以前一样

分布式事务（跨多个分片）：❌ 不建议
  跨多个 MySQL 实例 → 需要使用 XA/Seata 等分布式事务方案
  性能损失极大（2PC = 两阶段提交，网络交互多）

建议方案：
  尽量避免跨分片事务
  无法避免的 → 用 MQ + 本地事务 + 对账补偿（最终一致性）
```

### Q2：分库分表后怎么分页？

```
场景：后台管理员查所有订单（不知道 user_id）

解决：用 ES 做分页
  ① 所有订单数据同步到 ES
  ② 管理员在 ES 中搜索/分页
  ③ 需要详情时，用订单号查具体分片

为什么不用 MySQL 做：
  SELECT * FROM order ORDER BY create_time LIMIT 0, 20
  → 不知道 user_id → 需要查 8 个库 × 64 张表 × 前 20 条
  → 在应用层合并排序 → 内存消耗大，而且问题复杂

选 ES 的原因：
  ES 天然支持分布式分页
  ES 的搜索聚合功能比 MySQL 强大得多
```

### Q3：如果用户量从 1 亿涨到 10 亿怎么办？

```
扩容方案：

阶段一：增加从库（读不够）
  给每个分片增加更多的从库
  简单，不停机

阶段二：增加分片（写不够）
  从 8 库 64 表 → 16 库 128 表
  难点：旧数据在 8 个库中，需要迁移到 16 个库
  
  迁移方案：
  ① 双写（双倍写入旧库和新库）
  ② 定时任务迁移历史数据
  ③ 切换读流量到新库
  ④ 下线旧库
  
  一般不需要轻易增加分片，好的分片设计可支撑 3~5 年
```

### Q4：MySQL 和 ClickHouse 怎么选？

```
MySQL（行式存储）：
  ✅ 事务（ACID）
  ✅ 支持 UPDATE/DELETE
  ✅ 二级索引完整
  ❌ 大表聚合分析慢
  用途：在线交易（OLTP）

ClickHouse（列式存储）：
  ✅ 亿级数据聚合极快（毫秒级）
  ✅ 压缩率高（存 1TB 只占 200GB）
  ❌ 不支持事务
  ❌ 不支持 UPDATE/DELETE（有变通方案）
  用途：数据分析（OLAP）
```

