## 1. 系统概述

### 1.1 项目定位

构建一个面向 **海量数据 + 高并发** 场景的 B2C 电商购物平台数据库系统。系统需具备处理 **千万级商品 SKU、亿级注册用户、十亿级订单** 的能力。

### 1.2 核心业务指标

| 指标 | 目标值 |
|------|--------|
| 注册用户数 | ≥ 1 亿 |
| 在线商品 SKU | ≥ 1000 万 |
| 日订单量 | ≥ 500 万 |
| 秒杀峰值 QPS | ≥ 10 万 |
| 系统可用性 | ≥ 99.99% |
| API 响应时间 (P95) | ≤ 200ms |

### 1.3 数据库选型

| 数据库 | 版本 | 用途 | 说明 |
|--------|------|------|------|
| **MySQL** | 8.0 | OLTP 在线交易 | 主数据库，承载所有业务读写 |
| **Redis** | 7.x | 缓存/会话/计数器/分布式锁 | 热数据缓存，秒杀库存扣减 |
| **Elasticsearch** | 8.x | 搜索引擎 | 商品全文搜索，替代 MySQL LIKE |
| **ClickHouse** | 23.x | OLAP 冷数据存储 | 历史订单/评价归档，数据分析 |
| **MongoDB** | 7.x | 浏览历史 | 非核心大数据量存储（可选） |

### 1.4 中间件

| 中间件 | 版本 | 用途 |
|--------|------|------|
| **ShardingSphere-JDBC** | 5.x | 分库分表 + 读写分离路由 |
| **RocketMQ** | 5.x | 削峰填谷 + 异步解耦 + 数据同步 |

### 3.2 分片键设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                    分片键选择五大原则                          │
├─────────────────────────────────────────────────────────────┤
│ ① 查询即分片键 — 90%+ 查询场景应包含分片键，避免跨分片查询      │
│ ② 数据均匀分布 — 取模哈希保证分片间负载均衡                    │
│ ③ 关联同分片   — 主从表使用相同分片键，保证 JOIN 不跨分片      │
│ ④ 不可变键     — 分片键不应频繁变更（如按 user_id 而非 status） │
│ ⑤ 前置规划     — 提前确定分片数量，预留扩展空间                 │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 关键设计决策：订单按 user_id 分片

订单表是整个系统数据量最大、查询最频繁的表。分片键的选择是核心决策：

| 方案 | 查询分布 | 性能 | 用户查自己订单 | 按订单号查 | 结论 |
|------|---------|------|--------------|-----------|------|
| 按 order_id 分片 | 需广播8库 | 差 | 广播(慢8倍) | 直接命中 | ❌ 放弃 |
| **按 user_id 分片** | **命中率 90%+** | **优** | **单分片命中** | 需映射表 | ✅ **采用** |

**订单号→用户ID 映射方案**：`order_no_mapping` 表 + 订单号编码规则

## 5. 分布式 ID 方案

### 5.1 雪花算法 (Snowflake)

```
 0 | 0000000000 0000000000 0000000000 0000000000 00 | 0000000000 00 | 0000000000 00
 ↑                        ↑                              ↑                ↑
符号位                 41 位时间戳                     10 位机器 ID     12 位序列号
(1 bit)               (毫秒, 可用69年)                  (1024台机器)     (4096/毫秒)
```

| 字段 | 位数 | 说明 |
|------|------|------|
| 符号位 | 1 bit | 始终为 0，表示正数 |
| 时间戳 | 41 bit | 毫秒级，从 2024-01-01 起算，可用 69 年 |
| 机器 ID | 10 bit | 前5位=数据中心ID(32)，后5位=机器ID(32)，共1024台 |
| 序列号 | 12 bit | 每毫秒 4096 个，单机每秒 ≈ 400 万 ID |

### 5.2 各表 ID 策略

| 表 | ID 字段 | 生成方式 | 原因 |
|----|---------|---------|------|
| user | user_id | 雪花算法 | 分布式唯一用户ID |
| spu | spu_id | 雪花算法 | 商品唯一ID |
| sku | sku_id | 雪花算法 | SKU唯一ID |
| order | order_no | 雪花算法+业务编码 | 订单号需全局唯一且含业务信息 |
| cart | cart_id | 雪花算法 | |
| review | review_id | 雪花算法 | |
| coupon | coupon_id | 雪花算法 | |
| payment | payment_id | 雪花算法 | |
| refund | refund_id | 雪花算法 | |
| delivery | delivery_id | 雪花算法 | |

### 5.3 为什么不用自增ID？

```
MySQL AUTO_INCREMENT:
  × 分库分表后各分片自增值重复，无法保证全局唯一
  × 扩容时 ID 冲突风险

UUID:
  × 32位字符串，存储空间大
  × 无序 → B+树频繁页分裂 → 写入性能差

雪花算法:
  ✅ 有序递增（时间戳在高位）→ B+树末尾写入，页分裂少
  ✅ 全局唯一（时间戳 + 机器ID + 序列号）
  ✅ 高性能（纯内存计算，不依赖数据库）
  ✅ 64bit Long → 8 字节，存BIGINT
```

## 6. 索引设计策略

### 6.1 设计原则

| 原则 | 说明 |
|------|------|
| **覆盖索引优先** | 尽可能让查询走覆盖索引 (Using index)，避免回表 |
| **最左前缀匹配** | 联合索引遵守最左前缀规则 |
| **高区分度列在前** | 联合索引中区分度高的列放在前面 |
| **避免冗余索引** | `(a,b)` 已覆盖 `(a)` 的查询场景 |
| **控制索引数量** | 单表索引 ≤ 8 个，过多影响写入性能 |
| **索引下推** | MySQL 5.6+ ICP 特性可减少回表 |

### 6.2 核心索引清单

| 表 | 索引 | 类型 | 覆盖的查询场景 |
|----|------|------|-------------|
| user | `PRIMARY KEY (user_id)` | 聚簇索引 | ID 精确查询 |
| user | `uk_username` | 唯一索引 | 用户名登录 |
| user | `uk_phone` | 唯一索引 | 手机号登录/注册校验 |
| user | `uk_email` | 唯一索引 | 邮箱登录/注册校验 |
| user | `idx_register_time` | 普通索引 | 运营后台按注册时间筛选 |
| user_address | `idx_user_id (user_id)` | 普通索引 | 查询用户所有地址 |

## 8. Redis 缓存数据库设计

### 8.1 缓存架构

```
   请求
    │
    ▼
┌─────────────┐  命中率 ~98%   ┌─────────────┐  命中率 ~99%   ┌──────────┐
│ Caffeine    │ ←─────────── │ Redis       │ ←─────────── │ MySQL    │
│ 本地缓存     │  未命中        │ Cluster     │  未命中        │ (主库)    │
│ 0.1ms       │               │ ~1ms        │               │ ~5ms     │
└─────────────┘               └─────────────┘               └──────────┘
```

### 8.3 缓存更新策略

```
写入: 先更新 MySQL → 再删除 Redis (不是更新!)

读取: 先查 Redis → 未命中 → 互斥锁 → 查 MySQL → 写 Redis → 释放锁

删除 vs 更新：
  ✅ 删除缓存: 懒惰策略，读请求时才重建，简单可靠
  ❌ 更新缓存: 复杂，并发写入可能产生脏数据
```

详细 Redis 设计请参考 [Redis设计-第一期.md](./Redis设计-第一期.md)

## 9. 冷热分离策略

### 9.1 数据分层

```
             ┌─────────────────────────────────────┐
             │         热数据                       │
             │   MySQL (NVMe SSD)                   │
             │   最近 6 个月订单 / 最近 1 年评价     │
             │   数据量约 10%                       │
             │   承载 100% 读写                     │
             └──────────────┬──────────────────────┘
                            │ 每日 02:00 定时迁移
             ┌──────────────▼──────────────────────┐
             │         冷数据                       │
             │   ClickHouse (列式存储, HDD/云盘)     │
             │   历史订单 / 历史评价                 │
             │   数据量约 90%                       │
             │   存储成本仅 MySQL 的 1/5 ~ 1/10     │
             │   OLAP 分析性能远超 MySQL            │
             └──────────────────────────────────────┘
```

### 9.2 冷热分离规则

| 业务 | 热数据 | 冷数据 | 迁移条件 | 冷存储 |
|------|--------|--------|---------|--------|
| 订单 | 最近 6 个月 | 6个月+ 的已完结订单 | status IN (已完成/已取消/已退款) AND create_time < 6月 | ClickHouse |
| 评价 | 最近 1 年 | 1年+ 的评价 | create_time < 1年 | ClickHouse |
| 浏览历史 | 最近 1 个月 (MySQL) | 历史数据 | create_time < 1月 | MongoDB |
| 操作日志 | 最近 3 个月 | 3个月+ | create_time < 3月 | Elasticsearch |
| 购物车 | — | 7天未操作 | update_time < 7天 | 直接删除 |

## 10. 高并发应对方案

### 10.1 各场景并发矩阵

| 场景 | 并发等级 | QPS | 主要瓶颈 | 应对措施 |
|------|---------|-----|---------|---------|
| 商品浏览 | 高 | 50,000 | DB 读 + 带宽 | Redis缓存 + CDN + 读写分离 |
| 商品搜索 | 高 | 5,000 | MySQL LIKE | **Elasticsearch** 替代 LIKE |
| 加入购物车 | 中 | 5,000 | DB 写 | 异步写入 + Redis 缓冲 |
| 提交订单 | 高 | 10,000 | DB 写 + 一致性 | MQ 削峰 + 异步处理 |
| 支付回调 | 高 | 5,000 | DB 写 + 幂等 | Redis锁 + 幂等表 + 对账 |
| **秒杀** | **极高** | **100,000+** | **库存扣减** | **独立架构** (见下文) |
| 后台查询 | 低 | 100 | 大表扫描 | ES + ClickHouse 异构查询 |

### 10.2 秒杀独立架构

```
100,000 QPS
    │
    ▼
┌─────────────┐  过滤 90% 的无效请求
│ Nginx 限流    │  单 IP 每秒最多 5 次
└──────┬──────┘
       │  10,000 QPS
       ▼
┌─────────────┐  接口级别 QPS 硬限制
│ Sentinel 限流 │  秒杀接口独立限流池
└──────┬──────┘
       │  10,000 QPS
       ▼
┌─────────────┐  瞬时流量削峰为稳定消费
│ RocketMQ 削峰│  后端消费速率 1000/s
└──────┬──────┘
       │
       ▼
┌─────────────┐  原子校验 + 扣减
│ Redis LUA   │  库存校验 → 限购校验 → DECR
└──────┬──────┘
       │
       ▼
  返回秒杀结果 → 异步创建订单
```

### 10.3 库存扣减的两阶段提交

```
下单预扣阶段:
  Redis DECR product:stock:{sku_id} 1       ← 原子操作
  MySQL UPDATE sku_stock SET locked_stock = locked_stock + 1 WHERE version = ?

支付确认阶段:
  UPDATE sku_stock SET
    total_stock = total_stock - 1,
    locked_stock = locked_stock - 1,
    version = version + 1
  WHERE sku_id = ? AND version = ?

超时释放阶段 (30分钟后):
  Redis INCR product:stock:{sku_id} 1
  MQ 消息 → 释放 MySQL 预扣库存
```

## 11. 开发接入指南

### 11.1 ShardingSphere 配置

```yaml
# 订单库分片配置示例
spring:
  shardingsphere:
    datasource:
      names: order_db_0,order_db_1,order_db_2,order_db_3,order_db_4,order_db_5,order_db_6,order_db_7
    rules:
      sharding:
        tables:
          t_order:
            actualDataNodes: order_db_${0..7}.order_${0..63}
            databaseStrategy:
              standard:
                shardingColumn: user_id
                shardingAlgorithmName: order_db_inline
            tableStrategy:
              standard:
                shardingColumn: user_id
                shardingAlgorithmName: order_table_inline
            keyGenerateStrategy:
              column: order_no
              keyGeneratorName: snowflake
        shardingAlgorithms:
          order_db_inline:
            type: INLINE
            props:
              algorithm-expression: order_db_${user_id % 8}
          order_table_inline:
            type: INLINE
            props:
              algorithm-expression: order_${(user_id >> 4) % 64}
        keyGenerators:
          snowflake:
            type: SNOWFLAKE
```
